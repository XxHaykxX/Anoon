# anoon-admin — Вопросы к владельцу

> Собрано из gstack full-планирования (лензы CEO / Design / Eng / DX).
> Легенда: 🔴 блокер/критично · 🟡 нужно до старта · 🟢 вкус/дефолт (можно оставить рекомендацию).
> Формат ответа: номер → выбор. Где есть **Реком.** — если молчишь, пойду по ней.

---

## A. Стратегия и тайминг 🔴

**A1.** Существует ли уже v2-бэкенд (Next/Hono/Prisma) и есть ли живые пользователи, которых надо модерировать? Или строим админку под ещё несуществующее приложение?
> Все 4 лензы флагнули это как #1 блокер: в репозитории нет `server/`, схемы, модели `Report`. **Реком.:** сначала общая Prisma-схема, потом админка на моках, реальные данные — когда появится бэкенд+юзеры.

**A2.** По какому триггеру вообще начинать строить админку — число DAU, первая реальная жалоба, найм модератора? Назови порог. (Или «строим сейчас параллельно, несмотря на отсутствие бэкенда».)

**A3.** Сколько времени/бюджета готов вложить именно в анимации и визуальный полиш **внутреннего** инструмента (им пользуется 1–2 модератора, не клиенты)? **Реком.:** базовые микро-анимации да, тяжёлый полиш отложить.

---

## B. Приватность и юр 🔴 (для анонимного приложения — ядро риска)

**B1.** Может ли админ читать **личную переписку** анонимных пользователей?
- (a) Нет, никогда.
- (b) Только сообщения, привязанные к жалобе (+N контекста), с обязательным вводом причины и аудит-логом. **← Реком.**
- (c) Вся история.
> В PLAN это «открытый вопрос», в DESIGN уже заявлено как фича дровера — противоречие, решить до кода.

**B2.** Какие поля видит админ о юзере: только ник + #ID + счётчики, или ещё дата рег / страна / устройство? Email-hash и providerId **скрываем полностью**? **Реком.:** ник+#ID+счётчики; email-hash/providerId никогда не показывать.

**B3.** Юр-план на противоправный контент (в т.ч. CSAM) в медиа: обязанности по хранению, уведомлению, передаче правоохранителям — кто отвечает и по какому процессу? (нужен отдельный «escalate»-путь, блокирующий элемент вместо свободного просмотра).

**B4.** Где юридически находятся пользователи и сервер (РФ / ЕС / др.)? От этого зависят требования 152-ФЗ / GDPR к хранению медиа и логированию действий модератора.

**B5.** Срок хранения аудит-логов действий модератора + кто имеет доступ (только super-admin)?

**B6.** «Retention hold» на медиа, попавшее в жалобу — держим ли дольше обычного TTL, чтобы модератор успел разобрать? Сколько дней? **Реком.:** да, 30 дней.

## C. Продукт модерации (UX) 🟡

**C1.** Что модератор видит первым при входе — дашборд с метриками или **сразу очередь жалоб**? **Реком.:** очередь жалоб (двухпанельный triage), дашборд вторичен.

**C2.** Бан: обязательная **причина** (чипы+текст) и выбор **срока** (перманентный / временный / мут-до-даты) — или бан всегда перманентный? **Реком.:** причина обязательна + выбор срока.

**C3.** Нужны ли **массовые действия** (мультивыбор + «забанить выбранных») для волн спама/фейков? **Реком.:** да.

**C4.** Медиа при ревью: **блюр по умолчанию** + клик «Показать», видео стартует **с паузы и без звука**? **Реком.:** да (дейтинг-контент, защита модератора).

**C5.** Экран **«Журнал действий модератора»** (кто/кого/когда/почему) как первоклассная фича MVP? **Реком.:** да (приватность + комплаенс).

**C6.** Пустая очередь жалоб — нейтральное «жалоб нет» или позитивный «всё чисто» экран? (тон бренда). **Реком.:** нейтральное.

**C7.** Цвет «Мут/warning» `#F59E0B` визуально близок к бренд-жёлтому `#FDBF2D` — сместить warning в явно другой оттенок (оранж-красный/синий)? **Реком.:** да, сместить.

## D. Безопасность и доступ 🔴

**D1.** Admin-auth: **отдельная таблица `AdminUser`** (пароли argon2id) + отдельный ключ/audience JWT + **httpOnly-cookie** — согласен? (НЕ роль в общем пользовательском JWT — тот доверяет providerId, дыра). **Реком.:** да, отдельная система.

**D2.** **2FA (TOTP)** на вход админа — сразу или отложить? **Реком.:** сразу (доступ к PII).

**D3.** Роли (`super-admin` / `moderator`) закладываем в схему уже в MVP, даже если UI одноролевой? **Реком.:** да (ретрофит RBAC потом больно). Кто модератор на старте и сколько ожидается?

**D4.** Медиа из R2: **проксировать через admin-API** (аудит каждого просмотра, дороже) или presigned-URL напрямую в браузер (быстрее, но просмотр не залогируешь + enumeration)? **Реком.:** прокси.

## E. Тех / инфра / DX 🟡

**E1.** **Монорепо** (admin как подпапка `admin/` в этом репо, Vercel `Root Directory=admin`) или **отдельный репозиторий**? **Реком.:** монорепо-подпапка (проще шарить Prisma-схему).

**E2.** Как admin получает Prisma-схему и типы: общий импорт из `server/`, git submodule, или отдельный shared-пакет? **Реком.:** общий импорт из общего `packages/db` (когда появится).

**E3.** dataProvider: **REST через Next Route Handlers поверх общей Prisma сейчас** (репойнт на Hono позже, UI не меняется) — согласен? **Реком.:** да (держим сетевую границу).

**E4.** Начать с **mock/seed-провайдера** (`NEXT_PUBLIC_DATA_MODE=mock|api`), чтобы собрать и отполировать весь UI без бэкенда? **Реком.:** да.

**E5.** Скаффолд каркаса: `create-refine-app` (Next.js + headless + shadcn) или ручной `create-next-app` + доустановка? Пакетный менеджер + версия Node (pnpm? Node 22?). **Реком.:** create-refine-app headless + shadcn, pnpm, Node 22.

**E6.** Enforcement мут/бан: пассивный (сравнение `mutedUntil`/`expiresAt` при отправке на WS) + денормализованный счётчик для стат-карт — или фоновый reaper? **Реком.:** пассивный + денормализация.

**E7.** Деплой Vercel `Root Directory=admin` + preview-деплой на PR? Где `DATABASE_URL` для preview? **Реком.:** Vercel root=admin, preview на PR, отдельная preview-БД.

**E8.** Заводим `admin/.env.example` + `admin/README.md` (Setup / Env / Run mock / Run vs API / Deploy) уже на Этапе 1? **Реком.:** да.
